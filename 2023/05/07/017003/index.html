一、什么是API（应用程序接口）

Application Programming Interface（API），指的是一些软件或服务的接口，它们允许开发者通过一定的方式和规则来访问这些软件或服务的功能。你可以将API想象成一个服务窗口，你可以向窗口提出请求，获取你需要的信息或功能。比如，当你在使用一个手机App时，你通过API访问App的各种功能和信息，比如用户信息、位置信息、社交网络等等，而无需知道内部原理。

API拥有一系列好处，在我们长期的学习过程中，会有越来越深的体会。

二、传统模式与前后端分离模式对比

（一）传统开发模式

传统模式指的是将前端和后端应用程序集成在同一个应用程序中。在这种模式下，前端和后端共享同一个代码库和同一个服务器，前端通过模板引擎或者JSP等技术将数据渲染到页面上。这种模式的优点是开发和部署相对简单，但是由于前端和后端紧密耦合，当一个组件发生变化时，需要重新编译和部署整个应用程序，增加了维护成本。

（二）前后端分离模式

前后端分离模式则将前端和后端分别开发，前端通过API请求后端的数据并将其渲染到页面上。在这种模式下，前端和后端完全解耦，可以分别进行独立的开发和维护，而且前端开发人员可以专注于用户界面和用户体验，而后端开发人员可以专注于数据处理和业务逻辑。这种模式的优点是更容易扩展、更易于维护和更加灵活，但是需要更多的开发和部署工作。

前面都是一点点铺垫，了解一下常见的开发模式和API，下面开始正式介绍。

三、RESTful

（一）RESTful基本认知

RESTful（Representational State Transfer）是一种网络应用程序的设计风格和开发方式，基于 HTTP ，可以使用 XML 格式定义或 JSON 格式定义。

注意：RESTful只是一种接口设计风格。如果一个接口的url和参数符合REST原则，就称它为RESTful设计风格，如果一个接口的URL和参数不符合REST原则，就称它不符合RESTful设计风格。但是，不符合RESTful设计风格并不是错误的设计，对于一个后台接口来说，只要功能可以正常使用，URL和参数的设计没有固定法则。然而在实际开发中，RESTful风格已经得到了广泛使用。

（二）从名称上彻底理解RESTful

1.资源（Resources）

RESTful的中文是表现层状态转化。这里的主语被隐含了，实际上是资源，而在Web开发中这个资源指的就是URI。

所谓”资源”，就是网络上的一个实体，或者说是网络上的一个具体信息。它可以是一段文本、一张图片、一首歌曲、一种服务，总之就是一个具体的实在。你可以用一个URI（统一资源定位符）指向它，每种资源对应一个特定的URI。要获取这个资源，访问它的URI就可以，因此URI就成了每一个资源的地址或独一无二的识别符。

2.表现层（Representation）

“资源”是一种信息实体，它可以有多种外在表现形式。我们把”资源”具体呈现出来的形式，叫做它的”表现层”（Representation）。

比如，文本可以用txt格式表现，也可以用HTML格式、XML格式、JSON格式表现，甚至可以采用二进制格式；图片可以用JPG格式表现，也可以用PNG格式表现。

URI只代表资源的实体，不代表它的形式。严格地说，有些网址最后的”.html”后缀名是不必要的，因为这个后缀名表示格式，属于”表现层”范畴，而URI应该只代表”资源”的位置。它的具体表现形式，应该在HTTP请求的头信息中用Accept和Content-Type字段指定，这两个字段才是对”表现层”的描述。

3.状态转化（State Transfer）

访问一个网站，就代表了客户端和服务器的一个互动过程。在这个过程中，势必涉及到数据和状态的变化。

互联网通信协议HTTP协议，是一个无状态协议。这意味着，所有的状态都保存在服务器端。因此，如果客户端想要操作服务器，必须通过某种手段，让服务器端发生”状态转化”（State Transfer）。而这种转化是建立在表现层之上的，所以就是”表现层状态转化”。

客户端用到的手段，只能是HTTP协议。具体来说，常用的就是HTTP协议里面，五个表示操作方式的动词：GET、POST、PUT、PATCH、DELETE。它们分别对应四种基本操作：GET用来获取资源，POST用来新建资源（也可以用于更新资源），PUT用来更新资源，PATCH也用于更新资源（具体差异见后文），DELETE用来删除资源。

（三）Restful风格究竟是什么？

在请求层面，REST 规范可以简单粗暴抽象成以下两个规则：

  1. 请求 API 的 URI 表示用来定位资源；

  2. 请求的 METHOD 表示对这个资源进行的操作；

（四）具体规范

1.API请求的路径
  • 命名必须全部小写
  • 资源（resource）的命名必须是名词，并且必须是复数形式
  • 如果要使用连字符，建议使用‘-’而不是‘_’，‘_’字符可能会在某些浏览器或屏幕中被部分遮挡或完全隐藏
  • 易读

命名必须全部小写和易读都无需解释，可以理解为规定，那么为什么命名必须是名词且需要复数形式呢？这是因为在RESTful中，主语是资源，资源肯定是名词，不能是动词。其次，一个资源往往对应数据库中一张表，表就是实体的集合，因此需要是复数形式。

下面是一些反例：

  • api.example.com/getUser
  • api.example.com/addUser

下面是一些正例：

  • api.example.com/zoos
  • api.example.com/zoos/anim…
2.API请求的方法

对于如何操作资源，有相应的HTTP方法对应，常见的方法有如下五个（括号里表示SQL对应的命令）：

  • GET（SELECT）：从服务器取出资源（一项或多项）
  • POST（CREATE）：在服务器新建一个资源
  • PUT（UPDATE）：在服务器更新资源（客户端提供改变后的完整资源）
  • PATCH（UPDATE）：在服务器更新资源（客户端提供改变的属性）
  • DELETE（DELETE）：从服务器删除资源

示例：

HTTP方法 路径 表述
GET /zoos 获取所有动物园信息
POST /zoos 新建一个动物园
GET /zoos/ID 获取指定动物园的信息
PUT /zoos/ID 更新指定动物园的信息（前端提供该动物园的全部信息）
PATCH /zoos/ID 更新某个指定动物园的信息（提供该动物园改动部分的信息）
DELETE /zoos/ID 删除某个动物园
GET /zoos/ID/animals 获取某个动物园里面的所有动物信息

关于这些方法具体差异，这里暂时不作深入介绍。
