一、项目架构讲解

（一）业务（功能）架构

  • 顶层：用户服务，如注册登录、大会员权限、查找感兴趣视频等；
  • 中间层：在线视频流播放 + 实时弹幕；
  • 底层：管理后台，如视频上传、数据统计、系统消息推送等

其实我没看懂这是按什么标准分的……

（二）技术架构

  • 技术选型：SpringBoot2.x + MySql + MyBatis + Maven
  • 开发模式：项目采用经典MVC，控制层（Controller层）、服务层（Service层）、数据层（Dao层）

（三）部署架构

  • 前端：服务转发 + 负载均衡

服务转发：就是将前端接收到的请求转移给后端服务器；

负载均衡：当请求的数量比较大的时候，按照某种特定的规则，有规律地转发用户的请求，防止服务器负载过高影响效率甚至崩溃。

  • 后端：业务处理 + 功能实现
  • 工具：缓存、队列等

（三）实现热部署

根据chatgpt说法：

热部署是指在不重启服务器的情况下，修改代码后立即生效。虽然热部署在开发和测试阶段能够提高效率，但是它也存在一些坏处，包括：

  1. 系统稳定性：热部署过程中，应用程序和服务器的状态可能会出现不一致的情况，从而导致系统崩溃或无法正常工作。
  2. 安全性：热部署使得修改后的代码立即生效，这可能会导致系统中的安全漏洞被利用。
  3. 维护难度：热部署使得应用程序的代码处于不稳定的状态，这可能会导致在维护过程中出现意外错误或者难以排除的问题。
  4. 性能：热部署会增加系统的开销，因为在热部署过程中，需要重新加载类和资源，这会增加系统的负载和响应时间。

因此，在生产环境中，一般不建议使用热部署功能，而应该采用传统的重启方式来更新应用程序。

因此，本人决定不使用热部署。
